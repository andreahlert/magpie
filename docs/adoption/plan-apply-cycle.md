# Plan and apply cycle

Day-to-day workflow against an existing adoption.

## Vocabulary

- **intent** — `.apache-steward.intent.yaml` in the adopter repo. Source of truth. Committed. Edited by hand.
- **lock** — `.apache-steward.lock` in the adopter repo. Generated by `magpie apply`. Committed. Never edited by hand.
- **registry** — `registry/skills-index.json` in the framework snapshot. Read by the reconciler. Never edited.
- **plan** — the diff between the desired resolution (from the current intent + framework) and the on-disk lock.
- **apply** — write the resolved lock to disk and (optionally) materialise symlinks and render templates.

## The cycle

```
edit intent  ->  magpie plan  ->  read diff  ->  magpie apply  ->  commit
                                       |
                                       +--- iterate if the diff surprises you
```

## Typical commands

### See what would change

```bash
uv run --with pyyaml --with jsonschema --with jinja2 \
  python -m reconciler plan
```

Reads `.apache-steward.intent.yaml` and `.apache-steward.lock` from cwd. Exits 0 if nothing to do, 2 if changes are pending, 1 on error.

### Apply a change

```bash
uv run --with pyyaml --with jsonschema --with jinja2 \
  python -m reconciler apply \
    --symlink-target .claude/skills \
    --render-templates-to .apache-steward-overrides/rendered
```

Writes:

- the lock at `.apache-steward.lock`,
- one symlink per resolved skill under `.claude/skills/`,
- rendered templates under `.apache-steward-overrides/rendered/<skill-id>/`.

Idempotent. Re-running with no intent change is a no-op.

### Dry-run

```bash
... --dry-run
```

Computes everything, writes nothing. The stdout summary reports what would happen.

### Upgrade the framework

After `setup-steward upgrade` refreshes the snapshot:

```bash
uv run --with pyyaml --with jsonschema --with jinja2 python -m reconciler plan
```

If new skills landed in your declared domains, they appear as adds. New manifests may also change exclusion reasons for skills you previously had.

### When `apply` removes a skill

Plan rows with `-` mean the skill was in your previous lock but is no longer resolved. The detail field carries the reason:

```
- some-skill          intent.overrides.exclude
- another-skill       intent.capabilities.integrations
```

If the removal is intentional, apply. If not, edit the intent to bring the skill back (add the missing domain, integration, raise risk-tier-max, or `force-include`).

## Reviewing in PR

The committed pair (intent + lock) plus the rendered templates is what a reviewer sees. The diff tells the whole story:

- intent diff = what the adopter wants
- lock diff = how the reconciler resolved it
- rendered templates diff = what the agent will actually use

Reviewers can verify the lock matches the intent by running `magpie plan` locally and confirming "no changes".
